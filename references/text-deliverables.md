Day files hold source boundaries and working rules. Numbered folders hold final deliverables.

## Long-Form Rule

For whitepaper, Medium long article, brand manifesto, GTM, BP, proposal, or website structure:

1. Produce outline first.
2. Mark source-backed sections and `待确认` sections.
3. Wait for user confirmation unless the user explicitly asked for the full draft directly.
4. Then produce the full draft with public-risk checks.

## Brand Narrative Framework

| Section | Purpose |
| --- | --- |
| Project one-liner | Explain the project in one public-safe sentence |
| Category | Name the track without overclaiming |
| Audience | State who should care |
| Scene pain | Show the specific confusion, friction, or desire |
| Product role | State what the project changes |
| Narrative route | Choose operator clarity, technical trust, DeFi precision, IP world entry, or ecosystem institution |
| Proof layer | List source-backed assets, product screens, docs, roles, or confirmed events |
| Current stage | Say what is available or being prepared now |
| CTA | Use a stage-safe action |
| Risk boundary | State what is not public yet for internal review |

## Whitepaper Outline

```markdown
# [Project Name] Whitepaper

## 1. Project Overview
## 2. Market Or Category Context
## 3. Problem Statement
## 4. Product Or Ecosystem Positioning
## 5. User Segments And Use Cases
## 6. Product Modules Or System Architecture
## 7. Workflow Or User Journey
## 8. Brand And Community Layer
## 9. Token, Governance, RWA, Or Economic Layer
Only include if confirmed. Otherwise mark as 待确认 or omit from public version.
## 10. Roadmap
Only include dates and commitments when confirmed.
## 11. Risk, Safety, And Official Channels
## 12. Appendix: Source And Owner Confirmation Items
```

For public versions, remove internal mechanism claims that are not cleared.

## X Content

Use this structure for a launch or introduction thread:

| Tweet | Job |
| --- | --- |
| 1 | Project identity and clear category |
| 2 | Scene pain or market shift |
| 3 | What the project does |
| 4 | Why this matters for the target audience |
| 5 | Current stage and visible proof |
| 6 | Official update or safety line |
| 7 | Stage-safe CTA |

Short-post types:

| Type | Use |
| --- | --- |
| Opening post | First public account post or project introduction |
| Pain post | Explain the specific user problem |
| Product post | Explain one feature or module |
| Stage post | Clarify beta, event, release, or update status |
| Safety post | Official channel and anti-scam reminder |
| Visual caption | Pair with poster, long image, UI mockup, or character card |

X must avoid unsupported `airdrop`, `claim`, `rewards`, `now live`, `open registration`, `partner`, `audited`, `funded`, `guaranteed access`.

## Medium Or Long Article

Use this structure:

```markdown
# Title

## Opening
Name the reader and the scene.

## The Problem
Use one concrete Web3 workflow, market, community, or content problem.

## The Project
Explain what the project does now.

## Why This Direction Matters
Connect the project to the category without claiming leadership.

## Product Or Content System
Break into modules.

## Current Stage
State what is live, preparing, or pending.

## Official Updates And Safety
Keep official-channel and anti-scam line when community-facing.
```

## Community And AMA

Community welcome should include:

- What this community is for
- What users can expect now
- What is not yet confirmed
- Official-channel reminder
- Private key, seed phrase, password, verification code, and transfer safety line when relevant

AMA outline:

| Module | Content |
| --- | --- |
| Opening | Project identity and stage |
| Background | Why the project exists |
| Product or world | Main modules or story system |
| Current stage | What can be discussed now |
| User questions | Approved topics |
| Cannot answer | Token, rewards, financing, listing, partner, eligibility, wallet/data, legal status when not confirmed |
| Closing | Official update CTA |

## GTM And Commercial Plan

When the user asks for a commercial execution plan, include:

| Field | Required detail |
| --- | --- |
| Goal | Launch, waitlist, community activation, event attendance, ecosystem awareness, partner pipeline, content acquisition |
| Audience | Primary and secondary segments |
| Message | Route, one-liner, proof, CTA |
| Channel | X, Medium, Discord, Telegram, website, event, partner newsletter, KOL only if requested and allowed |
| Timeline | Week or day-level sequence |
| Assets | Copy, posters, long images, UI screenshots, deck, FAQ, AMA, community scripts |
| Roles | Founder, market, community, design, product, legal or owner review |
| KPI | Impressions, profile visits, CTR, qualified replies, community joins, AMA attendance, waitlist, content saves, partner calls |
| Risk gate | What must be reviewed before posting |

## Final Text Check

Before delivering text, check:

1. Does every major claim come from a source?
2. Does the stage match the source?
3. Does the copy avoid token, yield, partner, audit, funding, scale, and launch claims unless confirmed?
4. Does each platform have the right length and CTA?
5. Are `待确认` items left for owners instead of silently answered?
6. Does the tone match the category, not generic Web3 marketing?
