# Confidentiality And Public Release

Use this file whenever output may become public: README, GitHub repository text, examples, screenshots, public case studies, launch pages, public prompts, demo assets, or social posts.

## Redaction Gate

Before publishing, check and remove:

| Sensitive item | Public replacement |
| --- | --- |
| Client or project names without approval | `Case Alpha`, `Case Beta`, `Case Gamma`, or a track-based generic name |
| Local source paths | `current project source files` |
| Private filenames | `source material`, `brand assets`, `internal notes` |
| Internal screenshots | anonymized mockups or no screenshot |
| Partner, investor, audit, exchange, or grant names | `partner`, `investor`, `audit provider`, or remove the claim |
| Token mechanics, reward rules, whitelist details | `mechanism details pending owner approval` |
| Unreleased roadmap dates | `planned milestone` or `待确认` |
| Founder, team, or customer identity | role-based labels such as `founder`, `operator`, `community lead` |
| Wallet address, private data, dashboards | synthetic placeholders or no example |

## Public Repository Rules

1. Do not put real client names, source paths, source filenames, private screenshots, or unreleased facts in a public repository.
2. Do not publish examples that allow a reader to identify the client through a combination of track, visuals, copy, dates, and mechanism details.
3. Use anonymized cases only when explaining the method.
4. Use synthetic project names for examples, such as `OrbitOps`, `LiquidityOS`, `MythGate`, or neutral labels like `Case Alpha`.
5. Never mention that a public example is derived from a specific private customer project.
6. If a public repo already contains sensitive information, changing the latest file is not enough. Purge or rebuild history so old commits do not expose the material.

## Public Example Quality

A good public example should show the workflow without leaking the client:

| Example type | Safe content |
| --- | --- |
| Strategy sample | track, stage, target audience, source discipline, anonymized narrative route |
| Copy sample | invented project name, generic source ids, no private fact trail |
| Visual sample | synthetic UI, abstract components, no real logo, no private screenshot |
| UI sample | mock modules with neutral labels and placeholder data |
| Prompt sample | style and structure only, no client data or actual project facts |

## Final Public Check

Before release, answer:

1. Can a reader identify a client or project from the text?
2. Are there local paths or private filenames?
3. Are there unapproved claims about funding, partners, audits, tokens, launch status, user scale, TVL, APY, revenue, or roadmap dates?
4. Are images, screenshots, or UI examples synthetic or explicitly approved?
5. Does the public page present the Skill as a professional method rather than as a client case dump?

If any answer fails, redact before publishing.
