# Source Project Patterns

Use these anonymized patterns as calibration. They are abstracted production lessons, not client case studies. Do not copy their facts into a new project unless the current source files confirm them.

## Public Baseline

| Case | Project type | Public-safe center | Style anchor | Main risk lesson |
| --- | --- | --- | --- | --- |
| Case Alpha | Web3 SaaS, campaign entry, community operations | Clear user entry, official updates, small-scope preparation stage | Structured, official, low-density, operator workflow | Preparation-stage projects must not be written as live, open, fully launched, or reward-bearing. |
| Case Beta | DeFi, DEX, liquidity infrastructure with existing brand assets | Confirmed product identity, source-backed mechanism explanation, token-neutral user path | DeFi-native, compact, premium, product-led | Existing visuals may contain yield, fee, TVL, volume, bribe, or reward claims that need confirmation before reuse. |
| Case Gamma | Original IP, GameFi, events, merch, and community ecosystem | Characters, story world, content events, community participation | Character-centered, story-driven, vivid, non-financial first | Mixed IP and Web3 mechanism materials require a public story layer and a separate internal mechanism/risk layer. |

## Cross-Project Rules

1. Project facts and brand tone must come from current project files, not from these anonymized examples.
2. Day1 risk files are the production gate. Do not write public content before stage and risk are understood.
3. A public narrative should explain what the user can understand or do now. It should not promise token upside, yield, whitelist access, investor support, partnerships, audit status, user scale, or launch status without proof.
4. Visual design must express the current project's real category. Avoid default cyberpunk, purple gradients, trading screens, rockets, coin piles, and partner-logo walls unless the current VI supports them and the claim is verified.
5. Use `待确认` as a working category. Do not answer unknown facts with invented certainty.
6. Do not include client names, local paths, private source filenames, screenshots, partner names, unreleased token mechanics, or internal project facts in any public-facing example, README, screenshot caption, release note, or public repository file.

## Track Calibration

| Track | Narrative center | Suitable tone | Visual center | Frequent risks |
| --- | --- | --- | --- | --- |
| Infrastructure, L1/L2, protocol tooling | Reliability, developer path, composability, ecosystem access | Technical, concise, proof-led | Architecture diagrams, module maps, terminal or dashboard UI | Overstating adoption, partners, audits, throughput, roadmap commitments |
| DeFi, DEX, liquidity | Self-custody, market access, liquidity design, fee mechanics | Native, precise, restrained | Product UI, liquidity routes, token-neutral system diagrams | Yield promises, APY, bribes, fee split claims, TVL or volume without source |
| GameFi and original IP | Characters, world, gameplay loop, community content | Story-driven, vivid, non-financial first | Character cards, world gateways, event and merch visuals | Token, rewards, marketplace, jackpot, asset appreciation, fake launch status |
| SocialFi and community | Identity, community action, creator or member graph | Human, direct, community-native | Profile systems, social cards, badges, channels | Guaranteed growth, user scale, financialized social rewards |
| NFT and collectibles | Provenance, collection story, utility boundary, ownership experience | Cultural, object-specific, low-hype | Collection grids, item detail pages, launch visuals | Floor price, investment language, fake scarcity, unauthorized IP |
| RWA or asset-backed projects | Asset process, ownership boundary, compliance proof | Conservative, evidence-heavy | Process maps, legal or asset proof checklists | Investment advice, stable return, asset backing without documentation |
| DAO and foundation | Governance scope, contribution path, treasury or ecosystem role | Institutional, community-stable | Governance flows, proposal lifecycle, ecosystem maps | Legal status, voting power, treasury claims, partner or grant claims |
| Web3 SaaS or operation tools | Team workflow, status clarity, official updates, campaign path | Operator-focused, clean, practical | Low-density product UI, entry hubs, workflow modules | Writing planned features as live features, wallet or data privacy claims |

## Narrative Route Selection

When offering routes, make each route distinct enough to guide production:

| Route | Use when | Output tendency |
| --- | --- | --- |
| Operator clarity | Project helps teams coordinate work, campaigns, communities, tasks, updates, or launches | Website copy, product UI, X thread, workflow diagrams |
| Technical trust | Project depends on architecture, infra, security, developer adoption, or chain-level design | Whitepaper, docs-style homepage, architecture visuals |
| DeFi precision | Project is a DEX, liquidity, ve model, staking, lending, or yield-adjacent product | Risk-clean DeFi copy, mechanism explanation, product UI |
| IP world entry | Project has characters, story, game, merch, event, or community culture | Character content, worldbuilding whitepaper, posters, long images |
| Ecosystem institution | Project is foundation, DAO, launchpad, grant program, or public ecosystem | Official X, release hub, governance and ecosystem pages |

## Style Warnings From Anonymized Cases

- Case Alpha shows that early-stage SaaS needs preparation language, official update CTA, and abstract UI when screenshots, wallet permissions, or live flows are not cleared.
- Case Beta shows that existing Web3-native assets can look commercially strong while still carrying unverified claims. Preserve style, recheck claims.
- Case Gamma shows that a project can contain both useful story assets and high-risk financial mechanisms. Lead with the stable story layer, not the mechanism layer.
